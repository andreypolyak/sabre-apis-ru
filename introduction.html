Документация

Dev Studio

Вся документация по работе с Sabre APIs представлена на портале для разработчиков Sabre Dev Studio.

Общая информация по работе с Sabre APIs доступна в разделе SOAP basics.

Перечень сервисов, разбитый по категориям, доступен в разделе SOAP APIs.

Для каждого сервиса на Dev Studio есть страница с описанием и примером запроса и ответа, а также страница с ресурсами, на которой представлены:

  • история изменения версий сервиса (Release Notes)
  • документация по запросу к сервису (Request Documentation)
  • документация по ответу от сервиса (Response Documentation)
  • примеры запросов и ответов (Samples)
  • ссылка на WSDL-файл (WSDL)
  • ссылка на XSD-схемы запросов и ответов (Schemas)

Обратите внимание на то, что для доступа к странице с ресурсами по сервису, необходимо пройти авторизацию в Sabre Dev Studio. Для получения данных для авторизации (учетной записи), обратитесь к вашему куратору в Sabre.

Format Finder

Информация по работе с терминалом Sabre доступна на ресурсе Format Finder.

Для доступа требуется авторизация: референция (EPR), пароль и PCC.

Agency eServices

Информация по продуктам Sabre и новости Sabre доступны на агентском порталу Agency eServices.

Для доступа требуется авторизация: референция (EPR), пароль и PCC.

Сервисы Sabre

В Sabre существуют два вида сервисов: SOAP и REST. В данном руководстве описано взаимодействие только с SOAP сервисами.

Протокол SOAP

SOAP — протокол обмена сообщениями между несколькими системами, который использует язык разметки XML для сообщений.

Для взаимодействия с SOAP сервисами Sabre необходимо отправить специальным образом структурированное сообщение в формате XML через протокол HTTPS. В ответ сервис Sabre направит сообщение с результатами выполнения запроса. Все исходящие от сервисов Sabre сообщения всегда должны быть инициированы запросом со стороны системы агентства.

Для шифрования данных при отправке запросов к сервисам Sabre и получения ответов от них используется криптографический протокол TLS версии 1.2 и выше.

При работе с SOAP сервисами необходимо помнить, что они являются Stateful т.е. хранящими свое состояние. Это означает, что при отправке последующего запроса его ответ будет зависеть от предыдущего запроса. Например, если вы отправили запрос на добавление пассажира в бронирование, а потом отправили запрос на сохранение бронирования, то в бронировании будет один пассажир. Если же после запрос на добавление пассажира будет отправлен запрос на игнорирование, то пассажир не будет сохранен в бронировании.

Сообщения

Сообщения, отправляемые с использованием SOAP сервисов, должны быть обернуты в т.н. конверт (Envelope). SOAP конверт же должен содержать заголовок сообщения (Header) и его тело (Body):

<SOAP-ENV:Envelope xmlns:SOAP-ENV="http://schemas.xmlsoap.org/soap/envelope/" xmlns:eb="http://www.ebxml.org/namespaces/messageHeader" xmlns:xlink="http://www.w3.org/1999/xlink" xmlns:xsd="http://www.w3.org/1999/XMLSchema">
  <SOAP-ENV:Header>
    <!-- Заголовок сообщения-->
  </SOAP-ENV:Header>
  <SOAP-ENV:Body>
    <!-- Тело сообщения-->
  </SOAP-ENV:Body>
</SOAP-ENV:Envelope>

В заголовке сообщения необходимо указать следующие параметры:

  • ns4:MessageHeader/ns4:From/ns4:PartyId — идентификатор отправителя
  • ns4:MessageHeader/ns4:To/ns4:PartyId — идентификатор получателя (SWS)
  • ns4:MessageHeader/ns4:CPAId — PCC
  • ns4:MessageHeader/ns4:ConversationId — идентификатор диалога
  • ns4:MessageHeader/ns4:Service — название сервиса
  • ns4:MessageHeader/ns4:Action — название действия
  • ns4:MessageHeader/ns4:MessageData/ns4:MessageId — идентификатор сообщения
  • ns4:MessageHeader/ns4:MessageData/ns4:Timestamp — время отправки
  • wsse:Security/wsse:BinarySecurityToken — токен сессии или токен доступа (используется для всех запросов кроме запроса на создание сессии или токена доступа, подробнее см. раздел "Работа с сессиями")

Например:

<SOAP-ENV:Envelope xmlns:SOAP-ENV="http://schemas.xmlsoap.org/soap/envelope/" xmlns:eb="http://www.ebxml.org/namespaces/messageHeader" xmlns:xlink="http://www.w3.org/1999/xlink" xmlns:xsd="http://www.w3.org/1999/XMLSchema">
  <SOAP-ENV:Header>
    <ns4:MessageHeader xmlns:ns4="http://www.ebxml.org/namespaces/messageHeader" xmlns:ns5="http://www.w3.org/1999/xlink" xmlns:ns6="http://schemas.xmlsoap.org/ws/2002/12/secext" xmlns:ns7="http://www.opentravel.org/OTA/2003/05" xmlns:ns8="http://www.w3.org/2000/09/xmldsig#" xmlns:ns9="https://webservices.sabre.com/websvc">
      <ns4:From>
        <ns4:PartyId ns4:type="urn:x12.org:IO5:01">MyAagency</ns4:PartyId>
      </ns4:From>
      <ns4:To>
        <ns4:PartyId ns4:type="urn:x12.org:IO5:01">SWS</ns4:PartyId>
      </ns4:To>
      <ns4:CPAId>PCC</ns4:CPAId>
      <ns4:ConversationId>4e4ddad1-f897-4037-a7cf-4999c46dcd66</ns4:ConversationId>
      <ns4:Service ns4:type="sabreXML">EnhancedAirBookRQ</ns4:Service>
      <ns4:Action>EnhancedAirBookRQ</ns4:Action>
      <ns4:MessageData>
        <ns4:MessageId>6154f97a-b80c-4f78-a54b-eb6d962af738</ns4:MessageId>
        <ns4:Timestamp>2015-01-01T00:00:00.000</ns4:Timestamp>
      </ns4:MessageData>
    </ns4:MessageHeader>
    <wsse:Security xmlns:wsse="http://schemas.xmlsoap.org/ws/2002/12/secext">
      <wsse:BinarySecurityToken EncodingType="wsse:Base64Binary" valueType="String">Shared/IDL:IceSess\/SessMgr:1\.0.IDL/Common/!ICESMS\/ACPCRTC!ICESMSLB\/CRT.LB!-3441029479779067258!88802!0</wsse:BinarySecurityToken>
    </wsse:Security>
  </SOAP-ENV:Header>
  <SOAP-ENV:Body>
    <!-- Запрос -->
  </SOAP-ENV:Body>
</SOAP-ENV:Envelope>

Среды

В Sabre существует две среды для работы:

  • CERT. Тестовая среда для разработки и отладки приложений. Среда не подключена к системам перевозчика. Адрес для отправки запросов: https://sws3-crt.cert.sabre.com.
  • RES. "Боевая" среда для использования в приложениях. Среда подключена к системам перевозчика. Адрес для отправки запросов: https://webservices3.sabre.com.

Терминальные команды

Большинство сервисов Sabre имеет аналогичные команды, доступные в терминале Sabre (в синем экране). Соответствующую команду можно увидеть в документации к некоторым сервисам, а также в ответе на запрос, если в корневом элементе последнего указать значение параметра ReturnHostCommand="true". Такая возможность предусмотрена для тех сервисов, название которых содержит буквы LLS, например сервис IgnoreTransactionLLSRQ (игнорирование бронирования):

<IgnoreTransactionRQ ReturnHostCommand="true" xmlns="http://webservices.sabre.com/sabreXML/2011/10" xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" Version="2.0.0"/>

В ответе на данный запрос будет указана аналогичная команда для терминала Sabre — I:

<IgnoreTransactionRS Version="2.0.0" xmlns="http://webservices.sabre.com/sabreXML/2011/10" xmlns:stl="http://services.sabre.com/STL/v01" xmlns:xs="http://www.w3.org/2001/XMLSchema" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance">
   <stl:ApplicationResults status="Complete">
      <stl:Success timeStamp="2016-02-01T12:25:33-06:00">
         <stl:SystemSpecificResults>
            <stl:HostCommand LNIATA="193C0B">I</stl:HostCommand>
         </stl:SystemSpecificResults>
      </stl:Success>
   </stl:ApplicationResults>
</IgnoreTransactionRS>

Для работы с терминалом Sabre необходимо загрузить приложение Sabre Red Workspace.

Для доступа к тестовой среде (CERT) необходимо набрать в терминале команду ¤¤CERT, для перехода в боевую среду (RES) команду ¤¤RES. Для определения текущей среды команду *M.

Тестирование

Для тестирования отправки и приема запросов и ответов к сервисам, рекомендуется использовать бесплатное приложение SoapUI. Также доступен к загрузке файл с проектом SoapUI для работы с сервисами Sabre. Обратите внимание на то, что в проекте могут присутствовать не все сервисы, а используемые версии отставать от актуальных.
