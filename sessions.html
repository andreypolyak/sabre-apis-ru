Сессии

Сессии

Оглавление:

  • Создание сессии (SessionCreateRQ)
  • Обновление сессии (OTA_PingRQ)
  • Закрытие сессии (SessionCloseRQ)
  • Получение информации об используемых сессий (OLTMC USER DISP NAME)
  • Использование сессий, созданных в Sabre Red Workspace (OIATH)
  • Использование токенов доступа (TokenCreateRQ)

Для отправки любого запроса в Sabre Web Services требуется предварительное создание сессии. При создании производится авторизация в системе по имени пользователя (EPR, референция), паролю и PCC.

Сессия — аналог вкладки в Sabre Red Workspace. Одновременно в одной сессии может идти работа только с одним бронированием, при этом в разных сессиях может производиться параллельная работа над разными бронированиями. Максимальное количество одновременно открытых сессий определено в договоре.

Каждая сессия имеет максимальное время жизни — неделю. Также сессия имеет таймаут — 15 минут. Если за это время сессия не была ни разу использована, она будет принудительно закрыта.

Существуют две стратегии работы с сессиями:

  • Сессия создается перед отправкой любого запроса (или группы запросов) и закрывается после
  • Пул сессий. На стороне агентства существует брокер сессий, который всегда держит сессии открытыми, и по запросу распределяет сессии между запросами в Sabre

Рекомендации по работе с сессиями для обеих стратегий представлены в документе Connection Management: Best Practices and Strategies.

Обратите внимание на то, что каждое воскресенье в 00:15 по Центральноамериканскому времени (08:15 по Московскому времени летом и в 09:15 зимой) производится очистка всех сессий. В это время в течение приблизительно пяти минут запросы не будут отправляться, а все несохраненные данные будут удалены. Дополнительная информация об очистки сессий и рекомендации по работе с системой в это время доступны в документе Sabre APIs Pattern: SABRE SESSION HOUSEKEEPING (NORM OAA).

Подробнее о сессиях можно прочесть на Sabre Dev Studio.

Создание сессии (SessionCreateRQ)

Для создания сессии используется сервис SessionCreateRQ.

В запросе необходимо указать:

  • wsse:Username — имя пользователя (референция, EPR)
  • wsse:Password — пароль
  • Organization, Source/@PseudoCityCode — iPCC
  • Элемент Domain должен содержать значение DEFAULT

Ответ будет содержать токен созданной сессии wsse:BinarySecurityToken, который необходимо использовать при отправке других запросов.

Пример запроса

<SOAP-ENV:Envelope xmlns:SOAP-ENV="http://schemas.xmlsoap.org/soap/envelope/">
  <SOAP-ENV:Header>
    <MessageHeader xmlns="http://www.ebxml.org/namespaces/messageHeader">
      <From>
        <PartyId>Agency</PartyId>
      </From>
      <To>
        <PartyId>SWS</PartyId>
      </To>
      <ConversationId>b86df219-72e1-40aa-90f2-13f01ba055ee</ConversationId>
      <MessageData>
        <MessageId>176ac2fa-2f59-455e-816c-b38097588226</MessageId>
        <Timestamp>2016-02-01T07:32:09</Timestamp>
      </MessageData>
      <Action>SessionCreateRQ</Action>
    </MessageHeader>
    <Security xmlns="http://schemas.xmlsoap.org/ws/2002/12/secext">
      <UsernameToken>
        <Username>1234</Username>
        <Password>password</Password>
        <Organization>PCC</Organization>
        <Domain>DEFAULT</Domain>
      </UsernameToken>
    </Security>
  </SOAP-ENV:Header>
  <SOAP-ENV:Body>
    <SessionCreateRQ>
      <POS>
        <Source PseudoCityCode="PCC"/>
      </POS>
    </SessionCreateRQ>
  </SOAP-ENV:Body>
</SOAP-ENV:Envelope>

Пример ответа

<soap-env:Envelope xmlns:soap-env="http://schemas.xmlsoap.org/soap/envelope/">
  <soap-env:Header>
    <eb:MessageHeader eb:version="1.0" soap-env:mustUnderstand="1" xmlns:eb="http://www.ebxml.org/namespaces/messageHeader">

Обновление сессии (OTA_PingRQ)

Для того, чтобы сессия не была закрыта по таймауту необходимо отправить запрос к сервису OTA_PingRQ.

Пример запроса

<SOAP-ENV:Envelope xmlns:SOAP-ENV="http://schemas.xmlsoap.org/soap/envelope/">

    <MessageHeader xmlns="http://www.ebxml.org/namespaces/messageHeader">
      <From>

        <Timestamp>2016-05-23T12:54:10</Timestamp>
      </MessageData>
      <Action>OTA_PingRQ</Action>
    </MessageHeader>
    <Security xmlns="http://schemas.xmlsoap.org/ws/2002/12/secext">

  • POS/Source@PseudoCityCode — iPCC

Получение информации об используемых сессий (OLTMC USER DISP NAME)

Для того чтобы узнать количество используемых на данный момент сессий для определенного iPCC необходимо отправить терминальную команду OLTMC USER DISP NAME [iPCC] при помощи сервиса SabreCommandLLSRQ или в терминале Sabre Red Workspace.

Например, для получения информации о сессиях для iPCC 9LSC необходимо отправить команду OLTMC USER DISP NAME 9LSC.

Ответ будет содержать информацию о количестве используемых сессий (INUSE) и о максимальном количестве доступных сессий для данного iPCC (MAX):

Использование сессий, созданных в Sabre Red Workspace (OIATH)

Во время процесса разработки может быть полезно использование общего состояния, как при работе через Sabre APIs, так и при работе в терминале Sabre Red Workspace. Под общим состоянием подразумевается текущие несохраненные данные в бронировании, очередях и других объектах системы. Например, использование общего состояния позволяет добавить в бронирование сегменты при помощи сервиса EnhancedAirBookRQ без его сохранения, а затем увидеть эти сегменты в терминале Sabre Red Workspace, отправив команду *A.

Для этого необходимо в терминале Sabre Red Workspace отправить команду OIATH и получить в виде ответа токен сессии. Данный токен можно использовать при отправке запросов так же, как и токен созданный при помощи сервиса SessionCreateRQ.

Обратите внимание на то, что идентификатор сессии будет только та часть ответа, которая находится после ATH: в начале ответа и перед !X!E2E-1 в конце ответа (X — любая цифра).

Пример ответа:

ATH:Shared/IDL:IceSess\/SessMgr:1\\.0.IDL/Common/!ICESMS\/RESC!IC
ESMSLB\/RES.LB!-3364983643868498554!1481458!0!1!E2E-1

Идентификатор сессии из ответа:

Shared/IDL:IceSess\/SessMgr:1\.0.IDL/Common/!ICESMS\/RESC!ICESMSLB\/RES.LB!-3364983643868498554!1481458!0

Использование токенов доступа (TokenCreateRQ)

Токены доступа — альтернативный способ авторизации в Sabre. Отличие токенов доступа от сессий:

  • токены доступа могут быть использованы только для запросов к сервисам Bargain Finder Max
  • токены доступа имеют таймаут равный 1 неделе
  • токены доступа не могут быть закрыты или обновлены
  • максимальное количество открытых токенов доступа не ограничено
  • один токен доступа может быть использован для одновременной отправки нескольких запросов

Для создания токена доступа используется сервис TokenCreateRQ.

Ответ будет содержать токен доступа wsse:BinarySecurityToken, который необходимо использовать при отправке запросов к Bargain Finder Max.

      </To>

        <Timestamp>2016-02-01T08:09:22</Timestamp>
      </MessageData>
      <Action>TokenCreateRQ</Action>
    </MessageHeader>
    <Security xmlns="http://schemas.xmlsoap.org/ws/2002/12/secext">
      <UsernameToken>

        <Password>password</Password>
        <Organization>PCC</Organization>
        <Domain>DEFAULT</Domain>
      </UsernameToken>
    </Security>
  </SOAP-ENV:Header>
  <SOAP-ENV:Body>
    <TokenCreateRQ Version="1.0.0" xmlns="http://webservices.sabre.com"/>
