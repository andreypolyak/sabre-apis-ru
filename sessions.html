Рекомендации по работе с Sabre APIs

Сессии

Оглавление:

  • Создание сессии
  • Обновление сессии
  • Закрытие сессии
  • Использование токенов доступа

Для отправки любого запроса в Sabre Web Services требуется предварительное создание сессии. При создании производится авторизация в системе по имени пользователя (EPR, референция), паролю и PCC.

Сессия — аналог вкладки в Sabre Red Workspace. Одновременно в одной сессии может идти работа только с одним бронированием, при этом в разных сессиях может производиться параллельная работа над разными бронированиями. Максимальное количество одновременно открытых сессий определено в договоре.

Каждая сессия имеет максимальное время жизни — неделю. Также сессия имеет таймаут — 15 минут. Если за это время сессия не была ни разу использована, она будет принудительно закрыта.

Существуют две стратегии работы с сессиями:

  • Сессия создается перед отправкой любого запроса (или группы запросов) и закрывается после
  • Пул сессий. На стороне агентства существует брокер сессий, который всегда держит сессии открытыми, и по запросу распределяет сессии между запросами в Sabre

Рекомендации по работе с сессиями для обеих стратегий представлены в документе Connection Management: Best Practices and Strategies.

Обратите внимание на то, что каждое воскресенье в 00:15 по Центральноамериканскому времени (08:15 по Московскому времени) производится очистка всех сессий. В это время в течение приблизительно пяти минут запросы не будут отправляться, а все несохраненные данные будут удалены. Дополнительная информация об очистки сессий и рекомендации по работе с системой в это время доступны в документе Sabre APIs Pattern: SABRE SESSION HOUSEKEEPING (NORM OAA).

Подробнее о сессиях можно прочесть на Sabre Dev Studio.

Создание сессии

Для создания сессии используется сервис SessionCreateRQ.

В запросе необходимо указать:

  • wsse:Username — имя пользователя (референция, EPR)
  • wsse:Password — пароль
  • Organization, Source/@PseudoCityCode — iPCC
  • Элемент Domain должен содержать значение DEFAULT

Ответ будет содержать токен созданной сессии wsse:BinarySecurityToken, который необходимо использовать при отправке других запросов.

Пример запроса

<SOAP-ENV:Envelope xmlns:SOAP-ENV="http://schemas.xmlsoap.org/soap/envelope/" xmlns:eb="http://www.ebxml.org/namespaces/messageHeader" xmlns:xlink="http://www.w3.org/1999/xlink" xmlns:xsd="http://www.w3.org/1999/XMLSchema">
  <SOAP-ENV:Header>
    <ns4:MessageHeader xmlns:ns4="http://www.ebxml.org/namespaces/messageHeader" xmlns:ns5="http://www.w3.org/1999/xlink" xmlns:ns6="http://schemas.xmlsoap.org/ws/2002/12/secext" xmlns:ns7="http://www.opentravel.org/OTA/2003/05" xmlns:ns8="http://www.w3.org/2000/09/xmldsig#" xmlns:ns9="https://webservices.sabre.com/websvc">
      <ns4:From>
        <ns4:PartyId ns4:type="urn:x12.org:IO5:01">Agency</ns4:PartyId>
      </ns4:From>
      <ns4:To>
        <ns4:PartyId ns4:type="urn:x12.org:IO5:01">SWS</ns4:PartyId>
      </ns4:To>
      <ns4:CPAId>PCC</ns4:CPAId>
      <ns4:ConversationId>b86df219-72e1-40aa-90f2-13f01ba055ee</ns4:ConversationId>
      <ns4:Service ns4:type="sabreXML">SessionCreateRQ</ns4:Service>
      <ns4:Action>SessionCreateRQ</ns4:Action>
      <ns4:MessageData>
        <ns4:MessageId>176ac2fa-2f59-455e-816c-b38097588226</ns4:MessageId>
        <ns4:Timestamp>2016-02-01T07:32:09</ns4:Timestamp>
        <ns4:Timeout>60</ns4:Timeout>
      </ns4:MessageData>
    </ns4:MessageHeader>
    <wsse:Security xmlns:wsse="http://schemas.xmlsoap.org/ws/2002/12/secext" xmlns:wsu="http://schemas.xmlsoap.org/ws/2002/12/utility">
      <wsse:UsernameToken>
        <wsse:Username>1234</wsse:Username>
        <wsse:Password>PASSWORD</wsse:Password>
        <Organization>PCC</Organization>
        <Domain>DEFAULT</Domain>
      </wsse:UsernameToken>
    </wsse:Security>
  </SOAP-ENV:Header>
  <SOAP-ENV:Body>
    <SessionCreateRQ>
      <POS>
        <Source PseudoCityCode="PCC"/>
      </POS>
    </SessionCreateRQ>
  </SOAP-ENV:Body>
</SOAP-ENV:Envelope>

Использование токенов доступа

Токены доступа — альтернативный способ авторизации в Sabre. Отличие токенов доступа от сессий:

  • токены доступа могут быть использованы только для запросов к сервисам Bargain Finder Max
  • токены доступа имеют таймаут равный 1 неделе
  • токены доступа не могут быть закрыты или обновлены
  • максимальное количество открытых токенов доступа не ограничено
  • один токен доступа может быть использован для одновременной отправки нескольких запросов

Для создания токена доступа используется сервис TokenCreateRQ.

Ответ будет содержать токен доступа wsse:BinarySecurityToken, который необходимо использовать при отправке запросов к Bargain Finder Max.
