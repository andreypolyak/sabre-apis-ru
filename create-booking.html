Бронирование мест (EnhancedAirBookRQ)

Сервис EnhancedAirBookRQ является оберткой (orchestrated service) для сервисов OTA_AirBookLLSRQ, OTA_AirPriceLLSRQ и других.

Сервис последовательно выполняет все указанные в запросе подзапросы. Если установлен атрибут HaltOnError="true", сервис остановит выполнение подзапросов, в случае возникновения ошибок в них. Если установлен атрибут IgnoreOnError="true" произойдет игнорирование бронирования.

Рекомендуется установить атрибуты следующим образом:

  • HaltOnError="true"
  • IgnoreOnError="false" — вместо игнорирования бронирования рекомендуется воспользоваться логикой перебронирования (см. ниже).

Выбор сегментов

Информация о сегментах добавляется в элементы OriginDestinationInformation/FlightSegment:

  • FlightSegment/@ArrivalDateTime — дата прибытия
  • FlightSegment/@DepartureDateTime — дата отправления
  • FlightSegment/@FlightNumber и FlightSegment/MarketingAirline/@FlightNumber — номер рейса
  • FlightSegment/@DepartureDateTime — дата отправления
  • FlightSegment/@NumberInParty — количество запрашиваемых мест (но не пассажиров!)
  • FlightSegment/@ResBookDesigCode — класс бронирования
  • FlightSegment/@Status — статус сегмента (при бронировании используется стаус NN)
  • FlightSegment/DestinationLocation/@LocationCode — код пункта прибытия
  • FlightSegment/MarketingAirline/@Code — код маркетингового перевозчика
  • FlightSegment/MarriageGrp/@Ind — индикатор marriage сегмента (см. ниже)
  • FlightSegment/OriginLocation/@LocationCode — код пункта отправления

Например:

<FlightSegment ArrivalDateTime="03-15T09:20" DepartureDateTime="03-15T06:50" FlightNumber="1138" NumberInParty="003" ResBookDesigCode="Y" Status="NN">
  <DestinationLocation LocationCode="AER"/>
  <MarketingAirline Code="SU" FlightNumber="1138"/>
  <OriginLocation LocationCode="SVO"/>
</FlightSegment>

Значение MarriageGrp/@Ind для каждого сегмента формируется на основе информации о marriage сегментах из ответа Bargain Finder Max по следующему правилу:

Проверка статуса сегментов

Для OTA_AirBookLLSRQ необходимо задать список статусов сегментов, при получении которых в ответе считать, что произошла ошибка, взять места в запрашиваемом классе нельзя.

Рекомендуется указывать следующий список:

<HaltOnStatus Code="UC"/>
<HaltOnStatus Code="US"/>
<HaltOnStatus Code="NO"/>
<HaltOnStatus Code="NN"/>
<HaltOnStatus Code="LL"/>
<HaltOnStatus Code="UN"/>

На стороне Sabre статус сегментов будет проверяться NumAttempts раз (рекомендуемое значение 5), интервал между проверками WaitInterval миллисекунд (рекомендуемое значение 2000):

<RedisplayReservation NumAttempts="5" WaitInterval="2000"/>

Выбор валидирующего перевозчика

Код валидирующего перевозчика задается с помощью элемента PriceRequestInformation/OptionalQualifiers/FlightQualifiers/VendorPrefs/Airline. Например, <Airline Code="SU"/>.

Создание PQ

Атрибут PriceRequestInformation/@Retain="true" означает, что расчет будет сохранен, то есть в бронировании будут созданы PQ (Price Quotes). В случае если Retain="false", то PQ элементы созданы не будут. Рекомендуется установить значение Retain="true".

В элементе PriceRequestInformation/OptionalQualifiers/PricingQualifiers необходимо задать список типов пассажиров и указать их количество. Например, если в бронировании 2 взрослых, 1 ребенок и 1 младенец без места:

<PricingQualifiers>
  <PassengerType Code="ADT" Quantity="2"/>
  <PassengerType Code="CNN" Quantity="1"/>
  <PassengerType Code="INF" Quantity="1"/>
</PricingQualifiers>

Для каждого типа пассажира создается элемент PQ. В рассмотренном примере будет создано три элемента PQ, первый PQ для взрослых (ADT), второй для ребенка (CNN), третий для младенца без места (INF). Обратите внимание, что нумерация созданных PQ соответствует их порядку в элементе PricingQualifiers в запросе, это необходимо учитывать при создании связки пассажиров с PQ элементами.

Для перевозчиков, которые поддерживают Branded Fares, в случае выбора не самого дешевого бренда, требуется указать двухбуквенный код бренда в элементе /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/Brand.

Список кодов корпоративных скидок (Corporate id) задается в последовательно расположенных элементах /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/Corporate/ID.

Список аккаунт кодов (Account code) задается в последовательно расположенных элементах /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/Account/Code.

По умолчанию расчет стоимости производится по наиболее дешевым тарифам, доступным для указанных классов бронирования. Более дорогой класс может быть выбран при соблюдении одного или нескольких условий:

  • указан бренд тарифа
  • выбран расчет исключительно по приватным или публичным тарифам (значение true у атрибутов /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/FareOptions/@Private или /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/FareOptions/@Public)

В случае необходимости расчета по не самому дешевому тарифу (тарифам), требуется навязать код этого тарифа:

  • для одного тарифа — указать его код в атрибуте /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/CommandPricing/FareBasis/@Code
  • для нескольких тарифов — добавить несколько элементов /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/ItineraryOptions/SegmentSelect, у которых необходимо указать: номер элемента (@RPH), начальный (@Number) и конечный (@EndNumber) номера последовательности сегментов, для которых будет действовать указанный код тарифа, а также добавить элементы /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/CommandPricing/FareBasis, у которых должен быть номер элемента (@RPH), соответсвующий последовательности сегментов, и код тарифа (@Code).

Пре- и пост-процессинг

В запросе к сервису EnhancedAirBookRQ можно указать действия, которые следует выполнять до и после начала работы сервиса:

  • игнорирование бронирования до начала работы сервиса — требуется указать значение true для атрибута /EnhancedAirBookRQ/PreProcessing/@IgnoreBefore
  • игнорирование бронирования после завершения работы сервиса — требуется указать значение true для атрибута /EnhancedAirBookRQ/PostProcessing/@IgnoreAfter
  • просмотр бронирования после завершения работы сервиса — требуется добавить элемент /EnhancedAirBookRQ/PostProcessing/RedisplayReservation. Дополнительно в атрибуте /EnhancedAirBookRQ/PostProcessing/RedisplayReservation/@WaitInterval можно указать задержку в миллисекундах перед тем как будет запрошено чтение бронирования.

Добавление данных о пассажирах (PassengerDetailsRQ)

Сервис PassengerDetailsRQ является оберткой (orchestrated service) для сервисов, необходимых для создания бронирования. Сервис последовательно выполняет все указанные в запросе подзапросы. Если установлен атрибут HaltOnError="true", сервис остановит выполнение подзапросов, в случае возникновения ошибок в них. Если установлен атрибут IgnoreOnError="true" произойдет игнорирование бронирования.

Рекомендуется установить атрибуты следующим образом:

  • HaltOnError="true"
  • IgnoreOnError="true"

Добавление наземных сегментов

В случае, если пункт прибытия в одном из сегментов не совпадает с пунктом вылета следующего сегмента, необходимо добавить подзапрос ARUNK_RQ в запрос к сервису PassengerDetailsRQ. В этом случае в бронирование будут добавлены специальные наземные сегменты — ARUNK (Arrival Unknown).

Добавление пассажиров

Рассмотрим пример добавления в бронирования 2 взрослых, 1 ребенка и 1 младенца без места. Нумерация пассажиров в Sabre: 1.1, 2.1, 3.1, 4.1 и так далее.

<PersonName NameNumber="1.1" PassengerType="ADT">
  <GivenName>IVAN MR</GivenName>
  <Surname>IVANOV</Surname>
</PersonName>
<PersonName NameNumber="2.1" PassengerType="ADT">
  <GivenName>ELENA MS</GivenName>
  <Surname>IVANOVA</Surname>
</PersonName>
<PersonName NameNumber="3.1" NameReference="CHD" PassengerType="CNN">
  <GivenName>ANDREY</GivenName>
  <Surname>IVANOV</Surname>
</PersonName>
<PersonName Infant="true" NameNumber="4.1" PassengerType="INF">
  <GivenName>EKATERINA</GivenName>
  <Surname>IVANOVA</Surname>
</PersonName>

Обратите внимание на то, что максимальная длина полей имени и фамилии — 29 символов. Рекомендуется использовать следующий алгоритм заполнения полей:

Поле фамилии:

  1. заполняется фамилия
  2. если фамилия длиннее 29 символов, заполняются первые 29 символов фамилии

Поле имени (отчество не требуется):

  1. заполняется имя и титул
  2. если имя и титул длиннее 29 символов, заполняется только имя
  3. если имя длиннее 29 символов, заполняются первые 29 символов имени

Поле имени (требуется отчество):

  1. заполняется имя, отчество и титул
  2. если имя, отчество и титул длиннее 29 символов, заполняется имя и отчество
  3. если имя и отчество длиннее 29 символов, то заполняется имя, первая буква отчества и титул
  4. если имя, первая буква отчества и титул длиннее 29 символов, то заполняется имя и титул
  5. если имя и титул длиннее 29 символов, заполняется только имя
  6. если имя длиннее 29 символов, заполняются первые 29 символов имени

Для ребенка (от 2 до 12 лет) указывается атрибут NameReference="CHD". Возраст ребенка добавляется с помощью следующего элемента (SSR CHLD запрос) вложенного запроса SpecialServiceRQ:

<Service SSR_Code="CHLD">
  <PersonName NameNumber="3.1"/>
  <Text>15JAN10</Text>
  <VendorPrefs>
    <Airline Hosted="false"/>
  </VendorPrefs>
</Service>

где NameNumber — ссылка на номер взрослого пассажира, Text — дата рождения ребенка.

Для младенца без места указывается атрибут Infant="true". Для того чтобы младенец без места был ассоциирован с сопровождающим его взрослым необходимо во вложенном запросе SpecialServiceRQ указать (SSR INFT запрос):

<Service SSR_Code="INFT">
  <PersonName NameNumber="1.1"/>
  <Text>IVANOVA/EKATERINA/20FEB15</Text>
  <VendorPrefs>
    <Airline Hosted="false"/>
  </VendorPrefs>
</Service>

где NameNumber — ссылка на запись взрослого пассажира, Text — имя и фамилия (ограничение в 29 символов, см. выше) и дата рождения младенца.

Контактный телефон агентства (PhoneUseType="A") и контактный телефон клиента (PhoneUseType="M") должен быть добавлен с помощью элемента ContactNumbers/CustomerInfo/ContactNumber вложенного запроса TravelItineraryAddInfoRQ. Для каждого номера телефона также можно указать принадлежность к пассажиру(NameNumber="1.1").

<ContactNumbers>
  <ContactNumber Phone="74991234567" PhoneUseType="A"/>
  <ContactNumber NameNumber="1.1" Phone="79851234567" PhoneUseType="M"/>

E-mail адрес добавляется с помощью элемента CustomerInfo/Email этого же запроса TravelItineraryAddInfo. Для каждого адреса можно указать номер пассажира, которому принадлежит адрес (NameNumber="1.1") и тип адреса:

  • Type="TO" — адрес будет в поле To (Кому) письма
  • Type="FR" — адрес будет в поле From (От кого) письма
  • Type="CC" — адрес будет в поле CC (Копия) письма
  • Type="BC" — адрес будет в поле BC (Скрытая копия) письма

Например:

<Email Address="customer@customer.com" NameNumber="1.1" Type="TO"/>
<Email Address="agency@agency.com" Type="BC"/>

Обратите внимание, что в Sabre нет автоматической отмены бронирований (снятия мест) по таймлимиту. Поэтому на стороне агентства должен быть реализован процесс отмены бронирований по наступлению таймлимита (крайнего срока оплаты).

<EndTransactionRQ>

  <Source ReceivedFrom="SWS"/>
