Переход в другой PCC (ContextChangeLLSRQ)

Для создания бронирования в другом PCC необходимо перейти в него, отправив запрос к сервису ContextChangeLLSRQ. Запрос не потребуется, если бронирование нужно создать в iPCC или для создания бронирования будет использован сервис CreatePassengerNameRecordRQ.

Обратите внимание на то, что для перехода в другой PCC требуется наличие Branch Access между ним и iPCC, в котором была создана сессия. Подробнее о Branch Access см. в разделе Конфигурация Sabre.

В качестве значения атрибута /ContextChangeRQ/ChangeAAA/@PseudoCityCode в запросе указывается PCC, в котором должно быть создано бронирование.

Пример запроса

<ContextChangeRQ ReturnHostCommand="true" Version="2.0.3" xmlns="http://webservices.sabre.com/sabreXML/2011/10">
  <ChangeAAA PseudoCityCode="8XFG"/>
</ContextChangeRQ>

Бронирование мест (EnhancedAirBookRQ)

Сервис EnhancedAirBookRQ является оберткой (orchestrated service) для сервисов OTA_AirBookLLSRQ, OTA_AirPriceLLSRQ и других.

Сервис последовательно выполняет все указанные в запросе подзапросы. При этом, если установлено значение атрибута /EnhancedAirBookRQ/@HaltOnError равное true, сервис остановит выполнение подзапросов, в случае возникновения ошибок в них. Если значение атрибута /EnhancedAirBookRQ/@IgnoreOnError равно true, то произойдет игнорирование бронирования.

Рекомендуется установить в качестве значений обоих атрибутов true.

Выбор сегментов

Информация о сегментах добавляется в элементы OriginDestinationInformation/FlightSegment:

  • FlightSegment/@ArrivalDateTime — дата прибытия
  • FlightSegment/@DepartureDateTime — дата отправления
  • FlightSegment/@FlightNumber и FlightSegment/MarketingAirline/@FlightNumber — номер рейса
  • FlightSegment/@DepartureDateTime — дата отправления
  • FlightSegment/@NumberInParty — количество запрашиваемых мест (но не пассажиров!)
  • FlightSegment/@ResBookDesigCode — класс бронирования
  • FlightSegment/@Status — статус сегмента (при бронировании используется стаус NN)
  • FlightSegment/DestinationLocation/@LocationCode — код пункта прибытия
  • FlightSegment/MarketingAirline/@Code — код маркетингового перевозчика
  • FlightSegment/MarriageGrp/@Ind — индикатор соединенного (marriage) сегмента (см. ниже)
  • FlightSegment/OriginLocation/@LocationCode — код пункта отправления

Индикатор соединенного (marriage) сегмента может иметь следующие значения:

  • O — обычный сегмент
  • I — соединенный сегмент (продолжение предыдущего сегмента)

Например:

<FlightSegment ArrivalDateTime="03-15T09:20" DepartureDateTime="03-15T06:50" FlightNumber="1138" NumberInParty="003" ResBookDesigCode="Y" Status="NN">
  <DestinationLocation LocationCode="AER"/>
  <MarketingAirline Code="SU" FlightNumber="1138"/>
  <MarriageGrp>O</MarriageGrp>
  <OriginLocation LocationCode="SVO"/>
</FlightSegment>

Перебронирование в случае получения статуса сегмента UC

В некоторых случаях при попытке забронировать сегменты в тех классах бронирования, которые указаны в результатах поиска, перевозчик не подтверждает места. Неподтвержденные сегменты имеют статус UC и дальнейшая работа с ними невозможна.

Установка значения true у атрибута /EnhancedAirBookRQ/OTA_AirBookRQ/RetryRebook/@Option включает автоматическое перебронирование неподтвержденных сегментов по наиболее дешевым доступным классам бронирования.

Проверка статуса сегментов

Для OTA_AirBookLLSRQ необходимо задать список статусов сегментов, при получении которых в ответе считать, что произошла ошибка, взять места в запрашиваемом классе нельзя.

Рекомендуется указывать следующий список:

<HaltOnStatus Code="UN"/>
<HaltOnStatus Code="DB"/>
<HaltOnStatus Code="DN"/>
<HaltOnStatus Code="NN"/>
<HaltOnStatus Code="NO"/>
<HaltOnStatus Code="LL"/>
<HaltOnStatus Code="UC"/>
<HaltOnStatus Code="US"/>
<HaltOnStatus Code="HX"/>

На стороне Sabre статус сегментов будет проверяться NumAttempts раз (рекомендуемое значение 5), интервал между проверками WaitInterval миллисекунд (рекомендуемое значение 2000):

<RedisplayReservation NumAttempts="5" WaitInterval="2000"/>

Выбор валидирующего перевозчика

Код валидирующего перевозчика задается с помощью элемента PriceRequestInformation/OptionalQualifiers/FlightQualifiers/VendorPrefs/Airline. Например, <Airline Code="SU"/>.

Создание PQ

Атрибут PriceRequestInformation/@Retain="true" означает, что расчет будет сохранен, то есть в бронировании будут созданы PQ (Price Quotes). В случае если Retain="false", то PQ элементы созданы не будут. Рекомендуется установить значение Retain="true".

Категории пассажиров

В элементе PriceRequestInformation/OptionalQualifiers/PricingQualifiers необходимо задать список типов пассажиров и указать их количество. Например, если в бронировании 2 взрослых, 1 ребенок и 1 младенец без места:

<PricingQualifiers>
  <PassengerType Code="ADT" Quantity="2"/>
  <PassengerType Code="CNN" Quantity="1"/>
  <PassengerType Code="INF" Quantity="1"/>
</PricingQualifiers>

Для каждого типа пассажира будет создан свой собственный элемент PQ. В рассмотренном примере будет создано три элемента PQ: для взрослых пассажиров (ADT), для ребенка (CNN) и для младенца без места (INF). Обратите внимание, что нумерация созданных PQ соответствует их порядку в элементе PricingQualifiers в запросе. Это необходимо учитывать при создании связки пассажиров с PQ элементами.

Для бронирования перелета для младенца с местом необходимо использовать категорию пассажира INS.

Расчет по приватным тарифам

Список кодов корпоративных скидок (Corporate ID) задается в последовательно расположенных элементах /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/Corporate/ID.

Список аккаунт кодов (Account Code) задается в последовательно расположенных элементах /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/Account/Code.

Для расчета стоимости По умолчанию расчет стоимости производится по наиболее дешевым доступным тарифам. Однако, в запросе есть возможность указать необходимость расчета исключительно:

  • по публичным тарифам — значение true у атрибута /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/FareOptions/@Public
  • по приватным тарифам — значение true у атрибута /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/FareOptions/@Private

Пре- и пост-процессинг

В запросе к сервису EnhancedAirBookRQ можно указать действия, которые следует выполнять до и после начала работы сервиса:

  • игнорирование бронирования до начала работы сервиса — требуется указать значение true для атрибута /EnhancedAirBookRQ/PreProcessing/@IgnoreBefore
  • игнорирование бронирования после завершения работы сервиса — требуется указать значение true для атрибута /EnhancedAirBookRQ/PostProcessing/@IgnoreAfter
  • просмотр бронирования после завершения работы сервиса — требуется добавить элемент /EnhancedAirBookRQ/PostProcessing/RedisplayReservation. Дополнительно в атрибуте /EnhancedAirBookRQ/PostProcessing/RedisplayReservation/@WaitInterval можно указать задержку в миллисекундах перед тем как будет запрошено чтение бронирования.

Добавление данных о пассажирах (PassengerDetailsRQ)

Сервис PassengerDetailsRQ является оберткой (orchestrated service) для сервисов, необходимых для создания бронирования. Сервис последовательно выполняет все указанные в запросе подзапросы. Если установлен атрибут HaltOnError="true", сервис остановит выполнение подзапросов, в случае возникновения ошибок в них. Если установлен атрибут IgnoreOnError="true" произойдет игнорирование бронирования.

Рекомендуется установить атрибуты следующим образом:

  • HaltOnError="true"
  • IgnoreOnError="true"

Увеличение срока хранения бронирования в системе (Retention Segment)

По умолчанию Sabre хранит бронирование, по которому были осуществлены все перевозки, 15 дней. Для того чтобы увеличить этот срок можно добавить в бронирование специальный сегмент (Retention Segment) с датой, до которой необходимо хранить бронирование в системе. Для этого необходимо добавить элемент /PassengerDetailsRQ/MiscSegmentSellRQ, указав в нем:

  • /PassengerDetailsRQ/MiscSegmentSellRQ/MiscSegment/@DepartureDateTime — дата, до которой необходимо сохранять бронирование. Не более 360 дней от текущей.
  • /PassengerDetailsRQ/MiscSegmentSellRQ/MiscSegment/OriginLocation/@LocationCode — код города нахождения агентства, например MOW
  • /PassengerDetailsRQ/MiscSegmentSellRQ/MiscSegment/@NumberInParty — количество бронируемых мест для других сегментов.
  • /PassengerDetailsRQ/MiscSegmentSellRQ/MiscSegment/@Status — статус сегмента. Всегда GK.
  • /PassengerDetailsRQ/MiscSegmentSellRQ/MiscSegment/@Type — тип сегмента. Всегда OTH.
  • /PassengerDetailsRQ/MiscSegmentSellRQ/MiscSegment/Text — текстовое описание сегмента, например RETENTION.
  • /PassengerDetailsRQ/MiscSegmentSellRQ/MiscSegment/VendorPrefs/Airline/@Code — код перевозчика. Всегда YY.

Пример:

<MiscSegmentSellRQ>
  <MiscSegment DepartureDateTime="12-21" NumberInParty="1" Status="GK" Type="OTH">
    <OriginLocation LocationCode="MOW"/>
    <Text>RETENTION</Text>
    <VendorPrefs>
      <Airline Code="YY"/>
    </VendorPrefs>
  </MiscSegment>
</MiscSegmentSellRQ>

Добавление наземных сегментов

В случае, если пункт прибытия в одном из сегментов не совпадает с пунктом вылета следующего сегмента, необходимо добавить элемент /PassengerDetailsRQ/PostProcessing/ARUNK_RQ в запрос к сервису PassengerDetailsRQ. В этом случае в бронирование будут добавлены специальные наземные сегменты — ARUNK (Arrival Unknown).

Обратите внимание на то, что максимальная длина полей имени и фамилии — 29 символов. Рекомендуется использовать следующий алгоритм заполнения полей:

Поле фамилии:

  1. заполняется фамилия
  2. если фамилия длиннее 29 символов, заполняются первые 29 символов фамилии

Поле имени (отчество не требуется):

  1. заполняется имя и титул
  2. если имя и титул длиннее 29 символов, заполняется только имя
  3. если имя длиннее 29 символов, заполняются первые 29 символов имени

Поле имени (требуется отчество):

  1. заполняется имя, отчество и титул
  2. если имя, отчество и титул длиннее 29 символов, заполняется имя и отчество
  3. если имя и отчество длиннее 29 символов, то заполняется имя, первая буква отчества и титул
  4. если имя, первая буква отчества и титул длиннее 29 символов, то заполняется имя и титул
  5. если имя и титул длиннее 29 символов, заполняется только имя
  6. если имя длиннее 29 символов, заполняются первые 29 символов имени

Для ребенка (от 2 до 12 лет) указывается атрибут NameReference="CHD". Возраст ребенка добавляется с помощью следующего элемента (SSR CHLD запрос) вложенного запроса SpecialServiceRQ:

<Service SSR_Code="CHLD">
  <PersonName NameNumber="3.1"/>
  <Text>15JAN10</Text>
  <VendorPrefs>
    <Airline Hosted="false"/>
  </VendorPrefs>
</Service>

где NameNumber — ссылка на номер взрослого пассажира, Text — дата рождения ребенка.

Для младенца без места указывается атрибут Infant="true". Для того чтобы младенец без места был ассоциирован с сопровождающим его взрослым необходимо во вложенном запросе SpecialServiceRQ указать (SSR INFT запрос):

<Service SSR_Code="INFT">
  <PersonName NameNumber="1.1"/>
  <Text>IVANOVA/EKATERINA/20FEB16</Text>
  <VendorPrefs>
    <Airline Hosted="false"/>
  </VendorPrefs>
</Service>

где NameNumber — ссылка на запись взрослого пассажира, Text — имя и фамилия (ограничение в 29 символов, см. выше) и дата рождения младенца.

Для младенца с местом необходимо отправить аналогичный запрос, но в качестве номера пассажира указать его собственный номер, а не номер взрослого пассажира.

Добавление паспортных данных

Для отправки запроса SSR DOCS с номером паспорта необходимо создать элемент /PassengerDetailsRQ/SpecialReqDetails/SpecialServiceRQ/SpecialServiceInfo/AdvancePassenger.

В запросе должно быть указано:

  • AdvancePassenger/@SegmentNumber — номер сегмента, к кторому будет привязан SSR DOCS
  • AdvancePassenger/Document/@ExpirationDate — срок окончания действия договора
  • AdvancePassenger/Document/@Number — номер документа
  • AdvancePassenger/Document/@Type — тип документа (всегда указывается значение P)
  • AdvancePassenger/Document/IssueCountry — код страны выдачи документа
  • AdvancePassenger/Document/NationalityCountry — код страны рождения пассажира
  • AdvancePassenger/PersonName/@DateOfBirth — дата рождения пассажира
  • AdvancePassenger/PersonName/@DocumentHolder — индикатор владельца паспорта. Значение true указывается только в том случае, если в указанный документ в текущем бронировании вписаны другие пассажиры
  • AdvancePassenger/PersonName/@Gender — пол. Возможные значения:
    • M — мужской
    • F — женский
    • MI — младенец-мальчик
    • FI — младенец-девочка
  • AdvancePassenger/PersonName/@NameNumber — номер пассажира в бронировании. Для младенцев указывается номер сопровождающего его взрослого
  • AdvancePassenger/PersonName/GivenName — имя
  • AdvancePassenger/PersonName/MiddleName — отчество
  • AdvancePassenger/PersonName/Surname — фамилия

Пример SSR DOCS для взрослого:

<AdvancePassenger SegmentNumber="A">
  <Document ExpirationDate="2020-11-20" Number="1234567890" Type="P">
    <IssueCountry>RU</IssueCountry>
    <NationalityCountry>RU</NationalityCountry>
  </Document>
  <PersonName DateOfBirth="1980-11-20" DocumentHolder="true" Gender="M" NameNumber="1.1">
    <GivenName>IVAN</GivenName>
    <MiddleName>IVANOVICH</MiddleName>
    <Surname>IVANOV</Surname>
  </PersonName>
</AdvancePassenger>

Пример SSR DOCS для младенца (привязывается к сопроваждающему взрослому пассажиру с номером 1.1):

<AdvancePassenger SegmentNumber="A">
  <Document ExpirationDate="2017-04-15" Number="1234567890" Type="P">
    <IssueCountry>RU</IssueCountry>
    <NationalityCountry>RU</NationalityCountry>
  </Document>
  <PersonName DateOfBirth="2015-02-20" DocumentHolder="false" Gender="FI" NameNumber="1.1">
    <GivenName>EKATERINA</GivenName>
    <MiddleName>IVANOVNA</MiddleName>
    <Surname>IVANOVA</Surname>

Некоторые перевозчики требуют отправки дополнительных данных для идентификации пассажиров (SSR FOID). Для этого необходимо добавить элемент /PassengerDetailsRQ/SpecialReqDetails/SpecialServiceRQ/SpecialServiceInfo/Service со следующими данными:

  • Service/@SSR_Code — код (тип) SSR. Указывается значение FOID
  • Service/PersonName/NameNumber — номер пассажира в бронировании (для младенцев указывается их номер)
  • Service/Text — текст запроса в формате PP[код страны гражданства][номер паспорта]

Пример:

<Service SSR_Code="FOID">
  <PersonName NameNumber="1.1"/>
  <Text>PPRU1234567890</Text>
</Service>

Добавление контактной информации

Контактный телефон агентства (PhoneUseType="A") и контактный телефон клиента (PhoneUseType="M") должен быть добавлен с помощью элемента ContactNumbers/CustomerInfo/ContactNumber вложенного запроса TravelItineraryAddInfoRQ. Для каждого номера телефона также можно указать принадлежность к пассажиру(NameNumber="1.1").

<ContactNumbers>
  <ContactNumber Phone="74991234567" PhoneUseType="A"/>
  <ContactNumber NameNumber="1.1" Phone="79851234567" PhoneUseType="M"/>
</ContactNumbers>

E-mail адрес добавляется с помощью элемента CustomerInfo/Email этого же запроса TravelItineraryAddInfo. Для каждого адреса можно указать номер пассажира, которому принадлежит адрес (NameNumber="1.1") и тип адреса:

  • Type="TO" — адрес будет в поле To (Кому) письма
  • Type="FR" — адрес будет в поле From (От кого) письма
  • Type="CC" — адрес будет в поле CC (Копия) письма
  • Type="BC" — адрес будет в поле BC (Скрытая копия) письма

Например:

<Email Address="customer@customer.com" NameNumber="1.1" Type="TO"/>
<Email Address="agency@agency.com" Type="BC"/>

Ремарки — дополнительная текстовая информация, которая может быть добавлена в бронирование. Для добавления одной или нескольких ремарок в бронирование необходимо добавить один или несколько элементов /PassengerDetailsRQ/SpecialReqDetails/AddRemarkRQ/RemarkInfo/Remark.

Например:

<Remark Type="General">
  <Text>REMARK TEXT HERE</Text>
</Remark>

Отправка запроса специального сервиса (SSR) и дополнительной информации (OSI) перевозчику

Для отправки SSR или OSI необходимо добавить элемент /PassengerDetailsRQ/SpecialReqDetails/SpecialServiceRQ/SpecialServiceInfo/Service со следующими атрибутами и вложенными запросами:

  • /@SegmentNumber — номер сегмента в бронировании. В случае отсутствия атрибута запрос или дополнительная информация не будут "привязаны" к какому-либо сегменту
  • /@SSR_Code — код SSR из четырех букв или OSI для отправки дополнительной информации
  • /PersonName/@NameNumber — номер пассажира. В случае отсутствия атрибута запрос или дополнительная информация не будут "привязаны" к какому-либо пассажиру
  • /Text — текст запроса или дополнительной информации
  • /VendorPrefs/Airline/@Code — код перевозчика. В случае отсутствия атрибута запрос или дополнительная информация будут отправлены всем перевозчикам в бронировании
  • /VendorPrefs/Airline/@Hosted — значение false для всех перевозчиков, кроме American Airlines (значение true).

Пример запроса специального сервиса (SSR):

<Service SSR_Code="WCHR">
  <PersonName NameNumber="1.1"/>
  <Text>TEST SSR</Text>
  <VendorPrefs>
    <Airline Code="SU" Hosted="false"/>
  </VendorPrefs>
</Service>

Пример дополнительной информации (OSI):

<Service SSR_Code="OSI">
  <PersonName NameNumber="1.1"/>
  <Text>TEST OSI</Text>
  <VendorPrefs>
    <Airline Code="SU" Hosted="false"/>
  </VendorPrefs>
</Service>

Добавление бонусной карты (карты часто летающего пассажира)

Для добавления бонусной карты в запрос необходимо включить элемент /PassengerDetailsRQ/TravelItineraryAddInfoRQ/CustomerInfo/CustLoyalty со следующими атрибутами:

  • /@MembershipID — номер бонусной карты
  • /@NameNumber — номер пассажира
  • /@ProgramID — код перевозчика, выпустившего карту
  • /@SegmentNumber — номер сегмента в бронировании. В случае использования этого атрибута, не требуется указывать код маркетингового перевозчика
  • /@TravelingCarrierCode — код маркетингового перевозчика. В случае использования этого атрибута, не требуется указывать номер сегмента

Пример:

Отправка бронирования в очередь

Для отправки бронирования в очередь необходимо добавить элемент /PassengerDetailsRQ/PostProcessing/QueuePlaceRQ/QueueInfo/QueueIdentifier со следующими атрибутами:

  • /@Number — номер очереди
  • /@PrefatoryInstructionCode — PIC код (опционально)

Установка таймлимита

Значение таймлимита добавляется в бронирование с помощью элемента AgencyInfo/Ticketing вложенного запроса TravelItineraryAddInfoRQ, например:

<Ticketing TicketTimeLimit="09-20T23:00" TicketType="7TAW"/>

Обратите внимание на то, что значение таймлимита можно установить только с точностью до часа (22:00, 23:00 и т.д.).

Также в запросе можно указать PCC и номер очереди, в которую будет отправлено бронирование:

<Ticketing PseudoCityCode="IPCC" QueueNumber="55" TicketTimeLimit="09-20T23:59" TicketType="7TAW"/>

Значение таймлимита может быть получено из ответа на подзапрос OTA_AirPriceRQ сервиса EnhancedAirBookRQ, элемент PriceQuote/MiscInformation/HeaderInformation/LastTicketingDate. Например:

<OTA_AirPriceRS>
  <PriceQuote>
    <MiscInformation>
      <HeaderInformation>
        <LastTicketingDate>09-22T12:04</LastTicketingDate>
        <Text>VALIDATING CARRIER SPECIFIED - SU</Text>
        <Text>12DEC DEPARTURE DATE-----LAST DAY TO PURCHASE 22SEP</Text>
        <ValidatingCarrier Code="SU"/>
      </HeaderInformation>
    </MiscInformation>
    …

Обратите внимание, что в Sabre нет автоматической отмены бронирований (снятия мест) по таймлимиту. Поэтому на стороне агентства должен быть реализован процесс отмены бронирований по наступлению таймлимита (крайнего срока оплаты).

Создание бронирования в один запрос (CreatePassengerNameRecordRQ)

Для создания бронирования в один запрос может быть использован сервис CreatePassengerNameRecordRQ.
