Введение

При бронировании сегментов требуется получение подтверждения со стороны перевозчика. В некоторых случаях перевозчик не подтверждает забронированные сегменты, например, если последнее место в том классе бронирования, который был использован при бронировании сегментов, было недавно забронировано кем-то. Сегмент, для которого перевозчик не прислал подтверждение, получит соответствующий статус: UC, UN, LL или другой (см. полный перечень возможных статусов сегментов в Format Finder).

Для того чтобы не потерять клиента и все-таки предложить ему перелет выбранным им рейсом существует несколько различных вариантов обработки ситуации с неподтвержденными сегментами:

  • предложить клиенту перелет по другому (самому дешевому доступному) классу бронирования
  • предложить клиенту перелет выбранным им рейсом на другие даты

Отдельным образом следует обрабатывать ситуацию, при которой перевозчик не отвечает на запрос бронирования сегментов и их статус остается NN.

Перебронирование на самый дешевый доступный класс бронирования

Встроенный механизм перебронирования (EnhancedAirBookRQ, CreatePassengerNameRecordRQ)

Обратите внимание на то, что данный механизм производит перебронирование только в случае получения статуса сегмента UC. При получении других статусов сегментов, механизм не будет работать.

Сервисы EnhancedAirBookRQ и CreatePassengerNameRecordRQ имеют встроенный механизм для перебронирования на самый дешевый доступный класс бронирования тех сегментов, у которых был получен статус UC.

Для того чтобы активировать механизм перебронирования необходимо указать значение true у атрибута /EnhancedAirBookRQ/OTA_AirBookRQ/RetryRebook/@Option (для сервиса EnhancedAirBookRQ) или у атрибута /CreatePassengerNameRecordRQ/AirBook/RetryRebook/@Option (для сервиса CreatePassengerNameRecordRQ).

В случае, если было произведено автоматическое перебронирование сегментов, ответ на запрос будет содержать true в качестве значения атрибута /EnhancedAirBookRS/TravelItineraryReadRS/RetryRebook/@Successful (для сервиса EnhancedAirBookRQ). Для сервиса CreatePassengerNameRecordRQ необходимо сравнить значение атрибутов /CreatePassengerNameRecordRS/AirBook/OriginDestinationOption/FlightSegment/@ResBookDesigCode (статус при бронировании) и /CreatePassengerNameRecordRS/TravelItineraryRead/TravelItinerary/ItineraryInfo/ReservationItems/Item/FlightSegment/@ResBookDesigCode (статус после перебронирования).

Получение варианта перелета для перебронирования (EnhancedAirBookRQ)

В случае получения статуса сегмента, указанного в запросе к EnhancedAirBookRQ (значение атрибута /EnhancedAirBookRQ/OTA_AirBookRQ/HaltOnStatus/@Code) или CreatePassengerNameRecordRQ (значение атрибута /CreatePassengerNameRecordRQ/AirBook/HaltOnStatus/@Code) в качестве нежелательного, возможно отправить дополнительный запрос к сервису EnhancedAirBookRQ для получения варианта для перебронирования.

Запрос к сервису строится по аналогии с запросом на бронирование (см. Создание бронирований в 2 шага) за исключением:

  • указывается статус сегмента QF (атрибут /EnhancedAirBookRQ/OTA_AirBookRQ/OriginDestinationInformation/FlightSegment/@Status) — специальный статус сегмента, используемый исключительно для расчета стоимости. При бронировании сегментов с таким статусом не отправляется запрос в инвенторную систему перевозчика
  • отсутствует атрибут /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/@Retain — сохранение PQ не требуется
  • добавляется элемент /EnhancedAirBookRQ/OTA_AirPriceRQ/PriceRequestInformation/OptionalQualifiers/PricingQualifiers/BargainFinder — расчет стоимости по наиболее дешевым тарифам с учетом наличия мест
  • у атрибутов /EnhancedAirBookRQ/PreProcessing/@IgnoreBefore и /EnhancedAirBookRQ/PostProcessing/@IgnoreAfter устанавливается значение true — игнорирование текущего бронирования до и после выполнения запроса

В результате работы сервиса создаются фиктивные сегменты со статусом QF, для которых производится расчет стоимости по наиболее дешевым тарифам с учетом наличия мест (аналог терминальной команды WPNC), после чего бронирование игнорируется. Ответ на такой запрос будет содержать классы бронирования для указанных сегментов (/EnhancedAirBookRS/OTA_AirPriceRS/PriceQuote/MiscInformation/HeaderInformation/BargainFinder/AlternateBooking/@ResBookDesigCode, формат [номер сегмента][класс бронирования]), для которых есть места. В случае, если клиент соглашается на указанные тарифы, необходимо забронировать сегменты по новым классам бронирования. Бронирование производится так же как и при бронировании варианта, представленного в результатах поиска (см. Создание бронирований в 2 шага).

Обработка статуса сегмента NN (перевозчик не прислал ответ на запрос)

В некоторых случаях статус сегмента после выполнения запроса к сервису EnhancedAirBookRQ или CreatePassengerNameRecordRQ может остаться NN. Это означает, что перевозчик не прислал ответ на запрос бронирования сегмента.

Возможной причиной возникновения такой ситуации является то, что перевозчик не успел ответить на запрос в то время, которое было отведено на проверку обновления статуса сегмента в запросе к EnhancedAirBookRQ или CreatePassengerNameRecordRQ. В этих сервисах в запросе можно указать:

  • /EnhancedAirBookRQ/OTA_AirBookRQ/RedisplayReservation/@NumAttempts (/CreatePassengerNameRecordRQ/AirBook/RedisplayReservation/@NumAttempts) — количество проверок изменения статуса сегмента
  • /EnhancedAirBookRQ/OTA_AirBookRQ/RedisplayReservation/@WaitInterval (/CreatePassengerNameRecordRQ/AirBook/RedisplayReservation/@WaitInterval) — временной промежуток между проверками изменения статуса сегмента

В случае регулярного получения статуса сегмента NN рекомендуется увеличить указанные выше значения.

В остальных случаях возможна повторная отправка запроса на бронирование тех же сегментов, однако не рекомендуется повторять ее больше 2—3 раз.
